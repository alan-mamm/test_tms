# Item Manager — Тестовое задание

Fullstack-приложение для управления списком из 1 000 000 элементов.

## Стек

- **Бэкенд**: Node.js + Express.js
- **Фронтенд**: React 18, @dnd-kit (Drag&Drop)
- **Хранилище**: память сервера (in-memory, без БД)

## Функционал

### Левое окно (все элементы)
-  Фильтрация по ID (подстрока)
-  Инфинити-скролл порциями по 20
-  Добавление элементов с произвольным ID

### Правое окно (выбранные)
-  Фильтрация по ID
-  Drag&Drop сортировка (в т.ч. для отфильтрованного списка)
-  Инфинити-скролл порциями по 20
-  Состояние хранится на сервере (переживает обновление страницы)

### Очередь запросов
-  Дедупликация: одинаковые операции не отправляются дважды
-  Батчинг мутаций (select/deselect/reorder) — раз в **1 секунду**
-  Батчинг добавления элементов — раз в **10 секунд**

## Локальный запуск

```bash
# 1. Установить зависимости
npm run install:all

# 2. Собрать фронт
npm run build

# 3. Запустить сервер (отдаёт и API, и статику)
npm start
```

Приложение будет доступно на `http://localhost:3001`.

### Режим разработки (два терминала)

```bash
npm run dev:server

npm run dev:client
```

## Архитектурные решения

### Хранение 1 000 000 элементов
Элементы с ID 1–1 000 000 не хранятся явно — они существуют «по умолчанию».
Сервер хранит только два объекта: `selectedIds: number[]` и `extraIds: Set<number>`.
При запросе данные генерируются «на лету» за O(N) где N = 1 000 000 (-10 мс).

### Очередь с дедупликацией
Все мутации попадают в `Map<ключ, задача>`. Повторный вызов с тем же ключом
просто перезаписывает задачу — запрос к серверу уйдёт только один раз.

### Инфинити-скролл
Используется `IntersectionObserver` на «sentinel»-элементе (невидимый div
в конце списка). Как только он появляется в viewport — подгружается следующая
порция из 20 элементов.
